# BlackOut Trades — Production Readiness Audit (2026-07-04)

This report is the consolidated output of an automated multi-agent audit of the BlackOut Trades platform, covering seven dimensions: **data correctness**, **single source of truth**, **API/rate-limit safety**, **infra reliability**, **security**, **AI/BIE readiness**, and **product/UX**. Independent audit passes were run against each dimension and their findings were then cross-checked against one another and against the live codebase through adversarial verification — claims were re-traced to file:line evidence, deduplicated where multiple passes surfaced the same root cause, and downgraded or dropped where the evidence didn't hold up — before being assembled into the single verdict, findings list, and remediation plan below.

**Methodology note:** each dimension was audited independently by a dedicated pass reading the actual source (not documentation or self-report), producing findings backed by concrete file:line citations and, where applicable, reproduced evidence (query text, response payloads, duplicated logic diffs). A verification pass then adversarially re-checked every finding against the codebase before it was allowed into the final severity-ranked lists below, and reconciled overlapping findings across dimensions into the single set reported here.

Run as a 7-dimension, 18-agent background `Workflow` (pipeline → per-dimension verify → barrier merge → 3-way synthesis → assembly), per explicit CEO/CTO request. An interactive HTML version of this report (dashboard-style, with collapsible bug/fix entries) was also produced as a Claude Artifact for the same request.

**A note on internal arithmetic:** the source report's own "Totals" line states "21 findings across 19 distinct files," but the File-by-File table below lists 21 distinct file rows and the Exact Bug/Fix list below has 23 entries — a minor self-inconsistency in the assembly stage's count, not corrected here since this document reproduces the audit's own output faithfully rather than re-deriving it.

## Executive Summary

BLACKOUT is not ready for an unrestricted launch under the standard set — "every user-facing number accurate, traceable, fresh, and explainable" — but it is closer than a platform this complexity usually is, and the gap is narrow and well-defined rather than foundational. Seven independent audit passes converged on the same verdict: the correctness, rate-limiting, and infra engineering that already exists is frequently excellent — cluster-wide Redis rate limiters, build-time staleness flags, a real Layer-4 AI verifier, consistent ownership-scoped auth — but it is applied unevenly across the surface area. Two findings are true launch blockers because they directly contradict the trust promise in the platform's core value proposition, not edge cases: Largo streams every AI token to the user live, before its own numeric verifier ever runs, so a fabricated strike or premium is read and can be acted on before any check fires; and the 0DTE Command board hardcodes its freshness badge to "Live" regardless of whether the underlying scan actually produced data, so a silent upstream failure renders as a calm, reassuring, green "no plays right now" instead of a visible outage.

The single biggest risk theme across all seven dimensions is **inconsistent application of good patterns, not absence of them**. The same shape recurs everywhere: a canonical, well-designed guardrail exists (a verifier, a rate limiter, a rounding helper, a freshness computation) but 5–9 sibling call sites reimplement the logic by hand instead of importing it, and those siblings are the ones that drift, fail open, or ship ungrounded AI prose. This shows up as duplicated dollar-formatting logic that renders different numbers for the same premium, four independent WebSocket leader-locks that all fail open on Redis loss, and 7 of 9 real LLM call sites with no deterministic numeric guard despite the pattern existing elsewhere in the same codebase.

The single biggest strength is the reverse of that risk: where the pattern is applied, it's genuinely production-grade — Night Hawk's edition pipeline, the GEX heatmap's fabrication guard, and the cluster-wide rate limiters all already meet the bar this audit was measuring against, proving the team knows how to build this correctly and just needs to finish rolling it out.

**Recommendation:** hold wide launch for a focused sprint closing the 2 launch blockers and 10 high-severity items below (roughly 1–2 weeks of targeted work, not a re-architecture), then proceed.

## Top 20 Launch Blockers

Only 12 findings genuinely rise to launch-blocker or high severity — reporting exactly those rather than padding the list with medium/low items to reach 20.

1. **src/lib/largo-terminal.ts:433** — Largo streams every model token to the user live, and the Layer-4 numeric verifier only runs (and only appends a vague trailing caveat, never naming the bad numbers) after the full answer is already rendered. Fix: buffer high-stakes tool-loop answers so verification runs before any text reaches the client, and always surface `verification.unverified` in the `done` event. [ai_bie_readiness]
2. **src/components/zerodte/ZeroDteBoard.tsx:198** — The 0DTE Command board's freshness badge is hardcoded to `"Live"` and `as_of` is just response-build time, so a silently-failed scan renders as a calm, green, current empty board. Fix: derive freshness from a real upstream-success signal (mirroring `SpxSniperHeader`'s `resolveFreshness`) instead of a literal. [product_ux]
3. **src/lib/db.ts:119** — Documented production `PG_POOL_MAX=15` × 5 replicas (75 connections) oversubscribes PgBouncer's session-mode 20-connection backend budget by ~3.75x, and is the likely true cause of the still-ongoing "Query read timeout" storms a prior investigation misattributed using a wrong pool-size assumption. Fix: recompute `PG_POOL_MAX` against the documented 20-backend budget and wire pool-wait stats into an alert. [infra_reliability]
4. **src/lib/ws/uw-socket.ts:169** — All four WebSocket leader-election locks (UW, Polygon, options, stocks) fail open on any Redis error with zero gating on the confirmed 5-replica topology, risking a fleet-wide reconnect storm (code 1008) against upstreams that permit one connection per API key. Fix: gate fail-open behavior on `REPLICA_COUNT` (already used for this exact purpose elsewhere in the codebase) and fail closed to REST fallback when `REPLICA_COUNT>1`. [infra_reliability]
5. **src/lib/spx-play-claude.ts:347** — Claude's free-text thesis for real-money 0DTE play cards ships to members with no numeric grounding check, and the SPX Claude-gated play pipeline has zero durable audit-log write path for the verdict that gated the trade. Fix: run the shared numeric guard on `claude.thesis` before use, and add an audit-log write capturing `{verdict, confidence, thesis}` per play. [ai_bie_readiness]
6. **src/lib/providers/spx-commentary.ts:608** — The flagship SPX Live Desk AI commentary is returned and cluster-cached with no post-generation numeric grounding check, unlike the sibling `gex-heatmap/explain` route which already has one. Fix: extract and apply that route's `narrativeLevelsAreGrounded` guard here before caching. [ai_bie_readiness]
7. **src/lib/db.ts:3873** — `fetchNighthawkOutcomeAnalytics` uses bare Postgres `CURRENT_DATE` instead of the codebase's mandatory ET-date convention, a regression of a bug already found and fixed elsewhere, shifting Largo's and the admin track-record page's day window a day early between ~8pm–midnight ET. Fix: replace with `(NOW() AT TIME ZONE 'America/New_York')::date`, matching the established fix. [data_correctness]
8. **src/lib/api.ts:626** — The canonical dollar-formatter `fmtPremium` has 8 hand-copied siblings that render different strings for the same value (verified concrete divergences, e.g. $25,400 shows as both "$25K" and "$25.4K"), including in Largo tool output and Night Hawk LLM prompt text. Fix: extract to a shared module and delete the 8 reimplementations. [single_source_of_truth]
9. **src/lib/correctness/largo-verifier.ts:22** — Largo's tool-call *results* — the actual ground truth needed to trace a past answer's numbers back to source — are never persisted, only answer text and tool names, so a disputed answer can never be reconstructed after the fact. Fix: persist the already-built `capturedResults` alongside the answer. [ai_bie_readiness]
10. **src/lib/api-rate-quotas.ts:33** — The admin dashboard's vendor rate-limit headroom panel is computed from a per-replica in-memory counter while an adjacent, correct cluster-wide rollup exists in the same request, showing ops a false "ok" (~1/5th of real usage) right up to an actual rate-limit event. Fix: feed the panel from the already-computed `crossTelemetry` cluster rollup. [api_rate_limit_safety]
11. **src/components/desk/NetPremiumLeaderboard.tsx:51** — The component has no `loading` prop and renders its loading skeleton for any empty alert list, so a genuinely quiet tape period looks identical to a stuck/broken panel indefinitely. Fix: thread the existing `loading` state from the parent through and render a real empty-state message when loading is false. [product_ux]
12. **src/lib/et-date.ts:20** — The ET "today" calendar-date primitive is independently hand-copied in 6 places across two competing modules despite one explicitly claiming single-source-of-truth status; all 6 currently agree, but a future one-off DST/locale fix to only one copy would silently shift trading-session lock/settle boundaries. Fix: consolidate all 6 call sites on `et-date.ts`'s `todayEt()`. [single_source_of_truth]

## Top 20 Post-Launch Improvements

Fix: return a generic `db_unreachable` code publicly; keep the raw message in server-side logs only. [security]

Fix: apply the shared numeric guard to the critic's own `reason` output before it's allowed to cut/downgrade a play. [ai_bie_readiness]
3. **src/app/api/market/flow-brief/route.ts:158** — The 15-minute shared flow-brief memo is cached and served to every user with zero check that cited dollar figures/tickers match the actual alerts/dark-prints it was given. Fix: run the shared numeric guard before caching; cache null rather than a possibly-hallucinated line on failure. [ai_bie_readiness]
4. **src/lib/nights-watch/position-narrative.ts:158** — Per-position AI narratives are cached and served with no check that cited Greeks/levels/P&L match the position's actual data, enforced by prompt instruction only. Fix: apply the shared numeric guard before caching; fall back to the existing deterministic "what to do" logic on failure. [ai_bie_readiness]
5. **src/lib/nighthawk/play-explainer.ts:135** — "Full Hawk Intel" briefings only fall back to the deterministic, grounded explanation when generation fails outright — a fluent but hallucinated non-empty narrative passes through unchecked. Fix: apply the shared numeric guard to the success path too, not just the failure path. [ai_bie_readiness]
6. **src/lib/track-record-page.ts:154** — The internal track-record page and public share-card round the same win-rate fraction to different precision (one decimal vs. whole integer), and the platform's own page-vs-public consistency verifier doesn't check this field. Fix: route both through one shared percent-formatter and extend the verifier to assert the two agree. [single_source_of_truth]
7. **src/components/desk/FlowAlertStream.tsx:27** — The flow-alert tape's DTE fallback calculation uses local-time parsing and `Math.floor` instead of the canonical UTC-anchored `Math.round` math, undercounting by one day for some users across DST transitions. Fix: delete `calcDte` and import the canonical `daysToExpiry`. [single_source_of_truth]
8. **src/app/api/account/positions/[id]/detail/route.ts:44** — Night's Watch position-detail API skips the `roundFloats()` convention every sibling GEX route applies, so raw net-GEX values can reach the wire with float noise (currently masked by frontend `.toFixed()`). Fix: wrap the response in `roundFloats()` to match convention. [data_correctness]
9. **src/app/api/ready/route.ts:15** — The Railway healthcheck only pings Postgres; it never checks Redis, even though a `probeRedisHealth()` helper already exists (used only on an admin route), so a replica with dead Redis is marked fully healthy while rate limits, WS locks, and caches silently fail open. Fix: add a lightweight Redis ping surfaced as a non-fatal degraded flag. [infra_reliability]
10. **src/lib/spx-lotto-options.ts:171** — SPX Lotto's `spread_pct` is computed via raw float division with no rounding, and the `/api/market/lotto/today` response is never passed through `roundFloats()` (currently masked by UI `.toFixed()`). Fix: apply `roundFloats()` to the route's response body. [data_correctness]
11. **src/lib/bie/missed-alerts.ts:18** — The alert-producing cron list used for missed-alert reporting is a 3-entry hand-maintained constant that already omits `nighthawk-morning-confirm`, a cron that writes an identical member-visible status badge. Fix: derive the list from a field on the cron registry itself, or add a test asserting the two stay in sync. [infra_reliability]

## Data Integrity Architecture Proposal

The platform already has most of the right primitives — they're just applied inconsistently. This proposal is deliberately additive: formalize the pattern that `gex-heatmap`/`spx-desk`/Night Hawk already follow, extend it to the surfaces that skip it, and make freshness/violations first-class, queryable signals instead of implicit per-route conventions.

### The pipeline, as it should run at every hop

```
Vendor API (Polygon/UW)
   │  [CP-1: schema/shape guard]
   ▼
Backend transform (src/lib/providers/*, src/lib/*-engine.ts)
   │  [CP-2: numeric sanity + roundFloats()]
   ▼
DB / Redis cache (Postgres via src/lib/db.ts, Redis via lib/redis*)
   │  [CP-3: staleness stamp at write time]
   ▼
API response (src/app/api/**/route.ts)
   │  [CP-4: response-contract validation]
   ▼
Frontend display (src/components/**)
   │  [CP-5: freshness rendering, never fabricate]
```

#### CP-1 — Source ingestion guard
**Where it already exists:** `src/lib/providers/uw-rate-limiter.ts`, `polygon-rate-limiter.ts` gate *volume*; nothing today gates *shape*. **Extend:** add a thin schema-check wrapper at the point each provider client returns (`src/lib/providers/polygon-*.ts`, `src/lib/providers/uw-*.ts`) that rejects/quarantines a payload missing expected fields (e.g., UW snapshot missing `strike_totals`, Polygon aggregate missing `c`/`t`) before it ever reaches a transform. This is new work, but it's a single choke point per vendor client, not a redesign — it sits inside the existing provider modules.

#### CP-2 — Transform-time numeric sanity (the roundFloats() gap, generalized)
`src/lib/round-floats.ts` is the correct convention and is already applied at 17 boundaries. The audit found it's *missing* at two: `src/app/api/account/positions/[id]/detail/route.ts` (Night's Watch position detail) and `src/app/api/market/lotto/today/route.ts` (SPX Lotto `spread_pct`, computed in `src/lib/spx-lotto-options.ts:171/201`). Fix both by wrapping their `NextResponse.json(...)` bodies in `roundFloats(...)`, matching every sibling GEX/desk route. Then close the gap structurally, not just for these two: add a lint/test rule (`scripts/audit/`) that greps every `route.ts` under `src/app/api/market/**` and `src/app/api/account/**` for a `NextResponse.json` call not wrapped in `roundFloats`, and fails CI — the same way `scripts/audit/data-validator.mjs` already fails non-zero on data FAILs. This turns "we noticed float noise during an audit" into "CI catches it before merge."

#### CP-3 — Cache-write staleness stamping (extend the existing pattern, don't invent one)
The GEX heatmap / SPX desk / Night Hawk crons already compute `asof`/`stale` **at build time** (not serve time) — this is the correct pattern and should be the platform standard, not a per-pipeline choice. Concretely: any cache-warm path (`grid-warm`, `heatmap-warm`, `nights-watch-warm`, and the crons in `src/lib/cron-registry.ts` that aren't yet doing this) should write `{ data, asof, computed_at, upstream_ok }` as one object into Redis, never bare `data`. `upstream_ok` is the field that's currently missing from `ZeroDteBoard`'s pipeline (`src/lib/zerodte/scan.ts:116` swallows fetch failures via `.catch(() => [])`, then `route.ts:99` stamps `as_of: new Date().toISOString()` unconditionally) — that's the root cause of the hardcoded "Live" badge finding. The fix is mechanical once the convention is named: every warm-cron writes a real success/failure bit alongside its payload, and every route reads it back instead of synthesizing `as_of` from wall clock.

#### CP-4 — API response-contract validation
This is where `data-validator.mjs` currently lives, but it runs as an external, scheduled cross-check (Polygon/UW ground truth vs. member-facing numbers), not an inline gate. Keep it there for what it's good at (drift detection against vendor truth), but add a lightweight **inline** version at the route layer for the cheaper, structural checks: is `spread_pct`/`net_gex` a finite number, does `as_of` reflect `computed_at` from CP-3 rather than response-build time, does the `FreshnessChip` status the API sends match the actual data age. This closes the exact gap the `product_ux` findings surfaced (`ZeroDteBoard.tsx` hardcoding `status="live"`) — the API, not just the component, should refuse to claim "live" when `upstream_ok` is false.

#### CP-5 — Frontend display
`FreshnessChip` and the desk's `resolveFreshness()` (`SpxSniperHeader.tsx`) are the correct pattern — real age-derived status, not a hardcoded value. The fix is to make every consumer use them: `ZeroDteBoard.tsx` passing a literal `"live"` and `NetPremiumLeaderboard.tsx` conflating empty-with-zero-rows and loading are both violations of "never let the UI imply certainty the data doesn't have." Both are launch blockers already logged in `docs/audit/FINDINGS.md` and should be fixed as `fix/zerodte-freshness-badge` and `fix/net-premium-loading-state` per the standing branch policy.

### Where the data-correctness crons fit
`scripts/audit/data-validator.mjs` (Polygon+UW ground truth vs. served numbers, malformed-number scan) is the *external auditor* — it should keep running on the documented 13:32 UTC weekday cadence per `docs/audit/MARKET-OPEN-VALIDATION.md`, and its scope should grow to assert the CP-3/CP-4 invariants above (staleness field present and consistent, `roundFloats` applied — i.e., no `X.XXXXXXXXXXXXX` floats anywhere in a response body) rather than only ground-truth price/GEX comparison. `cron-staleness-watchdog` (already a real dead-man's-switch per the infra findings) is the *internal* half of the same job — it should be the thing that turns a CP-3 staleness stamp into an `admin_incidents` row (see next section) the moment a warm-cron's `computed_at` falls outside its expected cadence, rather than only detecting fully-dead crons.

## BIE Integration Plan

The CEO's directive is explicit and this plan does not soften it: **BIE ships first as a data auditor. It validates; it does not decide.** Every item below is BIE reading the platform's own numbers and reporting discrepancies — none of it touches `spx-play-engine.ts`, `spx-lotto-engine.ts`, conviction/score math, or any gate. This mirrors the standing Stage 6 boundary already documented in `docs/audit/FINDINGS.md`'s BIE-precedent-search entry ("never a live signal or a reason to change a gate/score") — this plan generalizes that same boundary from outcome-history lookups to numeric/data-integrity checks.

### What BIE-as-auditor validates

1. **Cross-surface consistency** — the exact class of bug the audit found live: `fmtPremium` vs. its 8 hand-copied siblings (`polygon-options-gex.ts`, `spx-commentary.ts`, `gex-positioning.ts`, `nighthawk/format.ts`, `nighthawk/grounding.ts`, `largo/flow-strike-stacks.ts`, `zerodte/intel.ts`) rendering different strings for the same dollar figure; the internal vs. public track-record win-rate rounding gap (`track-record-page.ts:154` vs. `track-record-public.ts:31`). BIE's job: periodically pull the same underlying number through every known rendering path and diff the strings, not just the raw values — this is a natural extension of the L4 verifier pattern in `src/lib/bie/verifier.ts`, redirected at the codebase's own outputs instead of at Largo's chat answers.
2. **Numeric grounding on every LLM narrative surface** — the audit found 7 of 9 LLM call sites ship prose with zero deterministic guard (`spx-commentary.ts`, `spx-play-claude.ts`, `nighthawk/play-explainer.ts`, `flow-brief/route.ts`, `nights-watch/position-narrative.ts`, `nighthawk/play-critic.ts`, and Largo's streamed tool-loop path in `largo-terminal.ts`). BIE's audit role here is to run the *already-built* grounding primitive — `narrativeLevelsAreGrounded()` from `gex-heatmap/explain/route.ts`, generalized into a shared helper — against each of these surfaces' output before or immediately after publish, and **report** mismatches. It does not rewrite the narrative or veto the play; it flags "this text cited a number not present in its source context" as a data-integrity signal.
3. **ET-date / staleness-window correctness** — the confirmed `db.ts:3873` bare `CURRENT_DATE` regression (identical bug class already fixed once in `confluence-outcomes.ts`) is exactly the kind of thing a scheduled BIE check should catch by re-running: query results computed with the ET-safe convention vs. the bare-`CURRENT_DATE` convention should be diffed nightly, and any drift flagged, so this bug class can never silently reappear a third time.
4. **Freshness-contract compliance** — using the CP-3 stamps described above, BIE checks that every route claiming `status: "live"` actually has a `computed_at` within the expected cadence window, catching the `ZeroDteBoard` class of issue at the data layer, continuously, not just via manual audit.
5. **Rate/pool telemetry accuracy** — the confirmed `api-rate-quotas.ts` bug (per-replica in-memory counter feeding a cluster-wide-looking dashboard stat, reading ~1/REPLICA_COUNT of true usage) is itself a data-integrity bug in the ops-facing surface. BIE auditing "does this dashboard's number match its own cluster-aware sibling number computed three lines away" is in-scope and cheap.

### Cadence

- **Every cache-warm cycle** (piggybacked on existing crons, not a new polling loop): CP-3 staleness-stamp validation and freshness-contract compliance (#4) — this is nearly free since it's just reading the stamp the warm-cron already writes.
- **Nightly (off-hours, batch)**: cross-surface consistency sweep (#1) and ET-date drift check (#3) — these are diff-and-report jobs, not latency-sensitive.
- **Per-publish, inline** (fits the existing pattern from `gex-heatmap/explain`): numeric grounding on LLM narratives (#2), for the highest-stakes surfaces first — `spx-play-claude.ts`'s thesis text (spliced into real-money 0DTE play cards, per the confirmed finding) and `spx-commentary.ts` before anything lower-stakes (position narrative, flow brief).
- **Weekly**: rate/pool telemetry accuracy (#5), since it's a slow-moving ops metric.

### How a violation is surfaced

Reuse what's already built rather than invent a new table:

- **`admin_incidents`** (`src/lib/admin-incidents.ts`) already has the right shape for this — `fingerprint`, `severity`, `category`, `title`, `detail`, `status: open/acked/resolved`, with dedup via `fingerprint = category:title` and MTTA tracking. BIE's data-integrity findings become `category: "data-integrity"` rows here, synced via a new `syncBieIncidents()` sibling to the existing `syncAdminIncidents()`, following the same open/ack/resolve lifecycle admins already use for `SpxAdminIssue`s. This gets BIE findings onto the existing admin dashboard for free, with no new UI.
- **A new `bie_audit_findings` table** (additive, not a replacement) for the *evidence* behind each incident — the specific number, the two surfaces that disagreed, the source context it failed to ground against. This is the durable trail the `ai_bie_readiness` audit found missing for exactly this kind of thing (Largo's `capturedResults` never persisted, per `src/lib/correctness/largo-verifier.ts`'s own header comment). `admin_incidents` stays the lightweight ops-facing pointer; `bie_audit_findings` is the forensic detail an engineer opens when triaging that incident.
- **Largo** is a *read* surface for this data, not a write one: extend Largo's existing tool surface (`src/lib/largo/tool-defs.ts`) with a `get_data_integrity_findings` tool, mirroring the already-shipped `get_similar_precedents` — a member or admin can ask "any known data issues on the GEX desk right now?" and Largo answers from `bie_audit_findings`/`admin_incidents`, explicitly historical/diagnostic color, never a trigger for any live behavior change.

### The boundary, restated explicitly

BIE-as-auditor reads: API responses, cache stamps, narrative text, telemetry counters. BIE-as-auditor writes: `admin_incidents` rows, `bie_audit_findings` rows, Largo-queryable summaries. **BIE-as-auditor never writes to**: `spx-play-engine.ts`, `spx-lotto-engine.ts`, `spx-power-hour-engine.ts`, `spx-play-claude.ts`'s approval gate, `play-critic.ts`'s verdict logic, or any conviction/score field. This is the same boundary already enforced for outcome-history lookups (`get_similar_precedents`'s tool description: "never a live signal or a reason to change a gate/score") — this plan extends it, doesn't create a new exception to it. No auto-tuning, no feedback loop into scoring, in this phase or any phase discussed here.
