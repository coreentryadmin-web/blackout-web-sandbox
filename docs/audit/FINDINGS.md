## рџџЎ INVESTIGATED 2026-07-03 15:4x UTC вЂ” "Query read timeout" storm across admin endpoints is real and ongoing, most likely caused by concurrent dual-agent audit load hitting a deliberately small per-replica connection pool, not a code defect
**Status:** INVESTIGATED, root cause understood, no speculative code change made. Pulled Railway `deploymentLogs` directly for the currently-live deployment (`c69b6c93`, live since 14:36 UTC) at the user's request to check build/deploy logs for issues вЂ” found **301 of the last 501 log lines were `error` severity**, dominated by `Query read timeout` across `admin/incidents`, `admin/health`, `admin-audit`, `admin/cron-health`, `admin/audit-log`, `admin/signal-analytics`, `api-telemetry-persist`, `cron/db-cleanup`, and the `[db] query failed while already reporting a failure` recursion-guard message (from PR #321). Timestamps span continuously from 15:24 to 15:43 UTC (essentially "now" at investigation time) вЂ” **this supersedes the earlier conclusion that the ~11:29-11:44 UTC burst was a resolved one-off blip: it is not resolved, it is ongoing.**

**What it is NOT:** not caused by the holiday-gate bug (PR #331) вЂ” that bug produced a false correctness *flag*, not a query failure, and the timeout pattern predates and postdates that fix. Not a "DB fully down" scenario вЂ” `/api/ready` (the member-facing critical-path health check) has responded in 0.2-0.6s on every check tonight, including during these bursts. The failures are concentrated entirely in **admin-panel/telemetry/cron-housekeeping endpoints**, never in member-facing `/api/market/*` reads.

**Most likely root cause (mechanical, from reading `db.ts`, not guessed):** the Postgres pool is deliberately small вЂ” `max: parseInt(process.env.PG_POOL_MAX ?? "5", 10)` **per replica** вЂ” with the comment explaining why: "PgBouncer sits in front of Postgres on Railway... we keep our own pool small... PgBouncer multiplexes these to many clients." This service runs **5 replicas** (`iad: 3, us-west2: 2` per the Railway service manifest), so up to 25 client connections can be in flight through a single shared PgBouncer at once. The connection string routes through `thomas.proxy.rlwy.net` (matches `connectionViaPooler()`'s `proxy.rlwy` check), so the server-enforced `statement_timeout` is skipped (PgBouncer rejects it as a startup param) and only the driver-side `query_timeout` (`PG_STATEMENT_TIMEOUT_MS` + 5000ms = 35s by default) actually bounds a query вЂ” a query merely *queued* waiting for a PgBouncer backend slot under load would eventually breach 35s and throw the exact generic "Query read timeout" seen everywhere, regardless of which specific query it was.

**Why tonight specifically:** this session and Cursor's session have both been running repeated full-audit scripts (`full-site-deep-audit.mjs`, `data-validator.mjs`-equivalent, `rth-browser-test.mjs`, direct `?force=1` cron probes) against the **same** admin/cron endpoints for hours, on top of each replica's own scheduled cron-health/incident checks. Two AI agents concurrently hammering the same small, shared connection pool is a load pattern that would not occur under normal single-admin-viewer usage вЂ” self-inflicted by the audit activity itself, not a pre-existing production bug from tonight's code changes.

**Why no code fix shipped:** bumping `PG_POOL_MAX` without visibility into PgBouncer's actual backend pool size to real Postgres could make contention worse, not better (more client-side connections competing for the same constrained backend slots) вЂ” that would be exactly the kind of unverified, speculative change the standing "never invent, verify before fixing" principle warns against. The responsible move right now is to (a) document the real mechanism, (b) stop being part of the problem, and (c) leave the actual pool/timeout tuning to a follow-up with real backend-connection visibility (Postgres `pg_stat_activity`, PgBouncer's own stats вЂ” none of which this sandbox can reach).

**Action taken:** throttling this session's own verification cadence going forward вЂ” spot-checking `/api/ready` for basic health rather than re-running the full `full-site-deep-audit.mjs` suite (which itself fires 3 concurrent cron probes) after every single deploy, reserving the full suite for genuinely significant changes. Flagged to Cursor (comment on PR #332) that their own tooling has direct DB access this sandbox lacks, in case they can pull `pg_stat_activity`/PgBouncer stats directly to confirm or refute the pool-contention hypothesis with real numbers.
